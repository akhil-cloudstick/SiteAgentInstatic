# Plan: SiteAgent v1 — minimal admin + full-Instatic tenants → Cloudflare
_The first shippable slice of the full plan ([`PLAN.md`](PLAN.md)). Scope set by the Operator. Locked via /grill-me-codex Act 1; Codex review log in [`PLAN-REVIEW-LOG.md`](PLAN-REVIEW-LOG.md)._

## v1 Goal
Stand up the smallest end-to-end loop: a **minimal Operator admin panel** that holds the AI key/model + Cloudflare creds and provisions tenants with a share link; each tenant gets a **full-featured** Instatic instance to build a site (by hand or via the AI agent); **Publish** ships the baked static site to the tenant's **Cloudflare Pages** project and shows the **live URL**. No team/roles, no usage dashboards, no billing — those come later.

## In scope (v1)
**Operator admin panel (minimal, Astro SSR) — exactly:**
- Set **OpenRouter API key + select model** (global).
- Save **Cloudflare credentials** (API token + account id).
- **Add tenant** → provision instance + Cloudflare project → return a **share login link + credentials** to hand to the tenant.

**Tenant (single Owner account = "Tenant Admin", FULL Instatic):**
- Open the share link, log in with the provided credentials.
- **ALL Instatic features — nothing held back:** add HTML / project, multiple pages, all data types, media, design system, drafts, the **AI agent** (routed through the **AI Gateway** — the key is hidden from the tenant), Super Import, forms-as-markup, etc.
- **Two creation paths** — *build new pages in Instatic* **or** *import & edit their own HTML/project* (Super Import) — each editable **two ways: by hand in the canvas, or by AI prompting** (the in-browser agent). Everything happens in the tenant's browser.
- **Preview** in Instatic's canvas.
- **Publish** → SiteAgent auto-deploys the baked output to the tenant's **Cloudflare Pages** → **live URL** shown.

## Out of scope (v1 → later versions)
- Tenant **sub-users / custom roles / permission toggles** (v1 tenant = single Owner; full roles UI is the next version).
- **Usage dashboards, metrics, health monitoring, alerts**; **AI per-tenant attribution / usage metering** (v1 ships the AI Gateway for **key-hiding only** — it forwards calls but does not meter/attribute per tenant yet).
- **Custom domains** (v1 = `*.pages.dev`), **billing**, fleet upgrade/backup UI.
- **Live form submissions** (forms can be built & published as markup; wiring submissions to a backend is deferred).

## v1 Architecture (trimmed from the full plan)
- **Operator Console** (`operator/`, Astro SSR): two screens — **Settings** (OpenRouter key/model + Cloudflare creds) and **Tenants** (add + show share link/creds + live URL/deploy status).
- **Provisioner (ordered, idempotent saga):** using the **single admin Postgres credential** (set once in env), per slug: **(1)** create the tenant's schema + a **least-privilege role** in the single shared DB — isolation **enforced by GRANT/REVOKE** (`REVOKE ALL ON SCHEMA public FROM PUBLIC`; grant the role `USAGE`/`CREATE` on **its own schema only**, never cross-grant); `search_path` is convenience, **not** the security boundary. The **admin/owner role** keeps cross-schema visibility (Operator sees all — no extra feature). **(2)** create `tenants/<slug>/uploads/` + generate `INSTATIC_SECRET_KEY` + the role password. **(3)** render per-tenant compose (`DATABASE_URL` → shared DB, tenant role, own schema). **(4)** `docker compose up -d`. **(5) headless first-run** (create the single Owner with **email + password only** — *no 2FA set by us*; the tenant enables their own 2FA after first login. Point the instance's OpenRouter **base URL at the AI Gateway** + set model — the **real key stays with admin; the tenant/AI never see it**. See spikes #2 and #4). **(6)** create the Cloudflare Pages project (namespaced **`siteagent-<slug>-<rand>`** — `<project>.pages.dev` is globally unique; store the returned URL). **(7)** record in the Registry. Each step is **check-then-act keyed by slug**; track state (`db_ready → up → seeded → cf_ready → done`); first-run is **guarded** (skip if Owner exists); **cleanup-on-failure walks back completed steps** (don't leak a scarce free-tier CF project).
- **Publish Deployer:** detect a *completed* bake reliably — publish to an **immutable `uploads/published/<timestamp>/`** and gate on an **explicit completion marker/manifest** (or an Instatic publish hook), **not** just `fs.watch` quiescence (`fs.watch` is unreliable cross-platform — esp. Windows dev vs Linux prod — and can fire mid-write) → `wrangler pages deploy <timestamp-dir> --project-name=siteagent-<slug>-<rand>` → record + show the live URL.
- **AI Gateway (control-plane, in v1 for key-hiding):** every tenant instance's OpenRouter base URL points here; the gateway **injects the single OpenRouter key — held only by admin** — and forwards to OpenRouter, so the **real key is never stored in any tenant instance** (tenant + AI can't read it). **Harden:** sign the per-tenant gateway token (validate on every call so a leaked URL can't be replayed/abused); add a **per-tenant rate guard** to cap runaway cost (no spend limit otherwise — one tenant could run up the bill through normal use); confirm AI calls run **server-side**, not from the browser. _Gated by spike #4; if Instatic rejects a custom base URL, force the instance's AI traffic through the gateway at the network level._ No per-tenant metering/dashboards in v1 (that's later). _(plan_review_2 #4/#5.)_
- **Instatic instance per tenant:** full features; own Postgres schema (in the single shared DB); **pinned image**.
- **Registry (Postgres `siteagent_control` schema in the same single DB), minimal:** `settings` (ai key/model, cloudflare creds), `tenants` (slug, owner-cred ref, cf project, ports, schema name), `deploys` (status, url).
- **Hosting:** v1 runs **entirely locally** — local Docker for the Instatic instance(s) + Operator Console, and a **locally-installed Postgres** (not containerized) as the single shared DB for dev/testing (containers reach it via `host.docker.internal` — set Postgres `listen_addresses='*'` + a `pg_hba.conf` rule allowing the Docker subnet with scram auth, else connections are refused); the developer plays Operator + tenant. Published sites deploy to **Cloudflare Pages** on the Operator's **free Cloudflare account** (`wrangler` Direct Upload) — real Cloudflare regardless. Deploy the same stack to **Linode** when onboarding real tenants (share links require the instance reachable).

## v1 build order
1. **Provisioner** happy path (Postgres schema+role in the single shared DB → container → first-run Owner (email+password) → point AI at the Gateway → Cloudflare project). _Gated by spike 9, 2, 4, 1, 3._
2. **Publish Deployer** (watch → `wrangler` Direct Upload → live URL).
3. **Operator Console** (Astro SSR): Settings + Tenants over the Registry + Provisioner.
4. **End-to-end:** add tenant → tenant builds + AI-edits → Publish → live on Cloudflare.

## v1 spike gates (prove first)
- **#1 Asset-URL portability** — baked HTML relative vs absolute (may force a small `publish.html` rewrite plugin into v1).
- **#2 Headless first-run** — create the Owner + inject the OpenRouter key/model. _Riskiest step — client demo confirms the admin API is cookie+CSRF, capability-gated, with TOTP step-up on some ops, so a clean REST provision is unlikely._ **First check for an official unattended bootstrap** (env vars / setup token, e.g. `INSTATIC_ADMIN_*`) before building anything. Else prove one of: **(a) encrypted DB seed** using `INSTATIC_SECRET_KEY`, or **(b) a one-shot headless-browser (Playwright) setup pass**. **Confirm the AI-key write is NOT TOTP-gated** (TOTP may apply only to import/replace). Both (a) and (b) are **version-coupled to pre-alpha** — pin one image, re-validate per upgrade. **v1 may instead do semi-manual first-run** (operator completes the wizard once in a browser) to de-risk the rest of the loop and defer full automation. The in-CMS AI agent is **browser-bridged** (29/35 tools need a live editor session) — fine since tenants edit in-browser, but it is **not** headless-drivable by us.
- **#3 Direct Upload renders** — `wrangler pages deploy` of the baked folder serves correctly.
- **#4 AI Gateway hides the key** — can we point Instatic's OpenRouter **base URL** at our gateway so the real key never lives in the tenant instance? If Instatic rejects a custom base URL, prove the **network-level fallback** (force the instance's outbound AI traffic through the gateway). Either way: tenant + AI cannot read the key.
- **#A Animation import fidelity** — import one motion/JS-heavy page, edit some text, publish; observe whether the animation still runs and whether the edit desyncs it. Decides the long-term approach for animated imports (lock sections vs separate tool vs rebuild).
- **#9 Schema-per-tenant works (PROVE FIRST — gates the whole DB design)** — stand up **two** Instatic containers against the **same database**, each on its **own schema**; confirm each runs its migrations **into its own schema, not `public`** (watch for hardcoded `public` or a *pooled* connection dropping `search_path`), creates **no objects/extensions in `public`** (pre-install any needed extensions as admin once), and the two **do not collide**. Pin `search_path` at the **role level** (`ALTER ROLE <role> SET search_path = <schema>`), not via the client connection (pooled connections lose client-set state — plan_review_2 #2). Verify isolation is **enforced by GRANT/REVOKE, not `search_path`**: after `REVOKE ALL ON SCHEMA public FROM PUBLIC`, tenant role A **cannot** `SELECT` from schema B *even fully-qualified*, and the admin role can read all. **Known limit (plan_review_2 #3):** Postgres **system catalogs are globally readable** — tenants can still see each other's *schema/table names + structure* (not row data); true metadata isolation needs DB-per-tenant or SQLite-per-instance (see DB-model decision). If Instatic hardcodes `public` or won't honor a per-connection schema → **fallback = DB-per-tenant on the same server** (off-the-shelf, guaranteed no collision).

## v1 Risks
- **DB layout (RESOLVED by Operator):** keep **schema-per-tenant** (one shared DB) with **spike #9 as a hard go/no-go gate before any provisioner code**; **DB-per-tenant on the same server is the documented fallback** if Instatic ignores a non-`public` schema. _(Self-review F3, resolved.)_
- **AI key hiding depends on the Gateway (spike #4)** — v1 keeps the ONE global key **only with admin**, behind the **AI Gateway**; it is **never stored in a tenant instance**, so neither the tenant nor their AI can read it. Residual: the gateway must stay reachable, and a tenant's *address-token* (not the real key) could leak — that only lets them call **our** gateway, which we can revoke/throttle. If Instatic won't accept a custom base URL, use the **network-level fallback** to force AI traffic through the gateway. _(F4 → resolved by the Gateway; no per-tenant keys, no spend cap.)_
- **Secrets are plaintext on the host (v1)** — per-tenant DB passwords + `INSTATIC_SECRET_KEY` live in `tenants/<slug>/`. Restrict file perms, gitignore `tenants/`; acceptable for **local v1 only** — plan a secrets manager before prod. _(F10.)_
- **Instatic is v0.0.6 pre-alpha** (confirmed by client demo) — APIs churn; no MCP, no content/import CLI; automation is the HTTP admin API only. Pin the image (freezes churn, but onto a buggy base) and budget for breakage on every upgrade. **Building a commercial fleet on a pre-alpha foundation is the headline strategic risk** — accept it knowingly for v1, or wait for a more stable tag.
- **Headless first-run + AI provider config** — *confirmed the hard part*: admin API is cookie+CSRF, capability-gated, with **TOTP 2FA step-up** on some ops; the AI agent is **browser-bridged**. Provisioning (create the Owner with email+password, point the AI provider at the **Gateway**) likely needs a **DB seed** (encrypted via `INSTATIC_SECRET_KEY`) or a **Playwright headless-browser first-run pass** — not a clean REST call. See spike #2.
- **Importing a tenant's own HTML** (Super Import): analysis is headless but **commit is browser-GUI** (fine — tenants work in-browser). **Motion-heavy HTML/JS is preserved as an opaque page-scoped script blob, not editable canvas nodes**, and canvas edits can desync from it. Plain HTML imports cleanly; **set tenant expectations for JS-animation-heavy sites.**
- **Brain-local in dev** ⇒ tenants can't reach their instance until it's on Linode; local v1 = the developer testing both roles.
- **Asset-URL portability** may pull the `publish.html` rewrite plugin into v1.
- **Postgres** adds setup vs SQLite; if v1 should be even leaner, **SQLite-per-instance** is a drop-in simplification (revisit if desired).
- **Schema-per-tenant hinges on spike #9** — the entire "one database, no collision" requirement depends on Instatic honoring a non-`public` schema; if it doesn't, we fall back to **DB-per-tenant on the same server**. Prove #9 before writing provisioner code.
- **Isolation is logical, not physical** — one shared database means a DB/server compromise (or the admin credential leaking) touches all tenants. Mitigated *between tenants* by per-tenant least-privilege roles (a tenant role is DB-blocked from other schemas); not mitigated against an admin/DB compromise. Acceptable for local v1 (sole operator); revisit for production.
- **Metadata leak ACCEPTED (Operator decision, plan_review_2 #3)** — tenants can read each other's *schema/table names + structure* via Postgres system catalogs (never the row data). **Harmless for v1** (sole operator). **Revisit before onboarding competing tenants** in production; the stronger-isolation upgrade paths stay documented: **SQLite-per-instance** (if Instatic supports it — bypasses #1/#2/#3 + blast radius) or **DB-per-tenant** (separate catalogs).
- **Only the admin credential is hand-set** (in env); per-tenant roles are auto-minted by the provisioner — keep that generation idempotent and ensure deprovision drops the role + schema cleanly.

## v1 Verification
- Add a tenant in the admin panel → instance boots, Owner login works (email+password), AI routed through the **Gateway** (key hidden, model set), Cloudflare project created, share link/creds returned.
- Log in as the tenant → build a page (by hand **and** via the AI agent) → preview in canvas → **Publish** → live URL renders correctly on Cloudflare (HTML/CSS/media).
- Re-run provisioning → idempotent; delete a tenant → instance + Postgres schema + CF project cleaned up (other tenants' schemas in the shared DB untouched).
